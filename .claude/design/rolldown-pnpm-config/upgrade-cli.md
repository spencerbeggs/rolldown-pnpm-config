# rolldown-pnpm-config - upgrade CLI

The `rolldown-pnpm-config upgrade` command that rewrites catalog version ranges in a config file in place, statically and surgically, with no execution of the config.

## Overview

`upgrade` keeps the catalog version ranges declared in a `PnpmConfigPlugin({...})` call current. It reads the config source, finds each catalog version literal, resolves available versions from the registry, lets the author choose per package (or applies latest-in-range non-interactively) and rewrites the version ranges in place at their byte spans. The versions stay visible in source; there is no separate generated manifest to reconcile.

This replaces the old `pnpm-plugin-silk` flow (track catalog deps in the root `package.json`, run `pnpm up -i -r`, regenerate). The CLI is a separate developer-facing subsystem from the build-to-runtime engine documented in [architecture.md](architecture.md); they share only the authoring shape (`PnpmConfigPlugin`'s inline `catalogs`) that `upgrade` reads and rewrites.

It lives entirely under `package/src/cli/` and is published via the `bin` entry `rolldown-pnpm-config` (`package/src/cli/bin.ts`). Reference this document when changing the upgrade pipeline, the peer recompute/drift/materialize behavior or the static discovery and rewrite mechanics.

## Pipeline

The command (`package/src/cli/commands/upgrade.ts`) wires single-purpose units in sequence. Data crosses each boundary as plain values.

`discover` (`discover.ts`) parses the source with `oxc-parser`, finds the one `PnpmConfigPlugin(...)` call and walks `catalogs.<name>.packages`. Each package whose range is a simple-operator string literal becomes a `CatalogEntry` carrying byte-offset spans for the range literal and any `peer` literal. Anything non-literal or with a complex multi-comparator range is returned in `skipped`, never rewritten.

`resolve` (`resolve.ts`) is the only Effect service: `RegistryResolver` shells out to `pnpm view <pkg> versions --json`, reusing the user's `.npmrc`, scoped registries and auth tokens. It also exposes `times` (`pnpm view <pkg> time --json`, publish dates feeding the release-age gate), `peerDependencies` (`pnpm view <pkg>@<version> peerDependencies --json`, per-version, feeding interop) and `pnpmConfig` (`pnpm config get <key>`, feeding the gate). Per-package failures are swallowed to a skip, never aborting the run.

The release-age gate filters every candidate list before `plan` or interop see it, so no path can propose a version pnpm would refuse to install. The gate model — combining partial contributions, the age filter and the `@pnpm/matcher` flat-string exclude semantics (`*` crosses `/`, deliberately not minimatch) — is the kit type `@effected/npm#ReleaseAgeGate`, a `Schema.Class` exposing readonly `ageMinutes`/`exclude`, static `combine`/`matchesExclude` and instance `isExcluded`/`filterVersions`. This repo owns only the two config readers in `release-age.ts` that turn its own sources into partial contributions. `computeGate` (`commands/upgrade.ts`) reads the `minimumReleaseAge`/`minimumReleaseAgeExclude` fields statically from the `PnpmConfigPlugin(...)` config (`readConfigReleaseAge`) and the pnpm-resolved values from `pnpm config get` (`parsePnpmGate`, its two independent reads run concurrently), each yielding a `PartialReleaseAgeGate | null`, then folds the non-null ones with the variadic total `ReleaseAgeGate.combine(...)`. The combined age is the strictest (max minutes, clamped ≥ 0, non-finite ignored) and the exempt set is the widest (union of patterns, insertion order preserved). `resolveGatedVersions` then fetches each package's versions plus `times` and calls the instance method `gate.filterVersions(versions, times, pkg, now)`, which drops any version younger than the age (and any with no publish timestamp) unless the package is excluded; an `ageMinutes` of 0 is a no-op, and the command reads `gate.ageMinutes` directly to skip the `times` fetch in that case.

`resolveGatedVersions` fetches the unique packages concurrently with bounded fan-out (`Effect.forEach` at `RESOLVE_CONCURRENCY = 12`) rather than serially, so a ~50-package config no longer pays ~100 sequential `pnpm view` spawns. Fail-closed semantics are unchanged per package: a versions-fetch `Left` yields `[]` and a `times`-fetch failure yields `{}` (every version then dropped as un-timestamped). It accepts an optional `onProgress(resolved, total)` callback; the command passes `writeResolveProgress` (a `Resolving N packages...` banner then an overwriting `Resolved X/N` counter on STDERR) only when `caps.interactive` (`cli/ui/env.ts`) is true, so CI/agent/non-TTY runs stay silent and stdout stays clean.

`resolveGatedVersions` returns `{ gated, raw, unresolved }`, where `unresolved` is the set of packages the registry could not resolve **at all** — a misspelt name, a package that no longer exists, an auth failure against a scoped registry. This is deliberately distinct from a package whose versions all fell to the release-age gate (`raw` non-empty, `gated` empty), which has a real publish history and is a legitimate "nothing old enough to offer yet", not an error. Before this distinction existed, a typo'd name failed its version fetch, was swallowed to an empty list the same way a fully-gated package is, planned to keep-only, counted as up to date by `buildWalkItems`, and was hidden from the table by the up-to-date filter below — the author never learned the name was wrong. `unresolved` now propagates to every surface: the interactive table warns in a banner above the rows (an unresolvable package has no row of its own to carry a per-row warning) and repeats the warning after the walk closes, since the in-table banner scrolls out of view once Ink tears down; `--preview` appends the same warning to its projection; `--yes` fails the run outright, writing nothing at all — not even the packages that did resolve, since an unattended run has nobody to read a warning and half-applying a config with a name that will never resolve is worse than applying none of it.

`plan` (`plan.ts`) computes the candidate list per entry with `@effected/semver`: latest-in-range (when newer than current), the `minor` tier (the latest within the current major line but beyond the caret range and below the overall latest, so a `^0.49.0` pin offers `0.50.0` rather than only a jump to the `1.0` major; emitted only when distinct from the in-range and latest picks), latest-overall (when newer than the in-range pick, possibly a major jump) and keep. Candidates are stable-only by default, with one exception: when an entry's own current range is itself a prerelease, candidates also include prereleases on the same named track (the leading prerelease identifier, e.g. `next`), so a `^3.0.0-next.8` pin can advance to `next.9` and beyond instead of freezing forever once `plan` stopped considering it "stable". A stable entry never sees a prerelease. When the entry carries a non-interop `strategy`, each non-keep candidate gets a recomputed `peerRange`; interop entries defer the peer to the group pass below.

The choice layer differs by mode. Non-interactive (`--yes`) takes latest-in-range directly and never the major. Interactive builds `WalkItem`s (`walk-plan.ts`) and runs the Ink `Walk` (`ui/Walk.ts`) driven by the pure `walk-reducer.ts` table state machine, collecting `Decision`s through the `runWalk` bridge (`ui/run-walk.ts`), which also threads the pre-built interop `GroupModel`s into the table and now resumes the fiber with `Effect.die` if Ink's `waitUntilExit()` rejects — so an Ink crash fails the run instead of hanging it suspended forever (issue #37). The table replaced an earlier per-package walk that stepped one entry at a time; see [The interactive table](#the-interactive-table).

Interop reconcile runs per catalog group, and differs by mode. Interactively it is live: before the walk, `buildGroupModel` (`interop-live.ts`) pre-fetches each interop candidate version's in-group peerDependencies, and `computeGroupPeers` then recomputes every member's peer floor and any conflict synchronously from the user's current picks — the table itself is the reconcile. The write path honors the final picks plus those live-derived floors directly (no post-walk downgrade or re-prompt) and reports any remaining conflicts. The `--yes`/CI path instead runs the batch `runInterop` reconcile (`interop.ts`). Either way, interop range and peer edits come from `buildInteropEdits` and these entries are excluded from the normal per-entry edit loop so spans never overlap. The group pipeline and its two modes are described under [Peer handling](#peer-handling).

### The interactive table

The interactive path shows every discovered package at once, grouped by catalog, instead of stepping through them one at a time — modelled on `pnpm up -i`. Every discovered row is shown, up-to-date rows included as non-selectable context; `initTable` starts the cursor on the first actionable row (falling back to row 0 when nothing is actionable) rather than filtering the inert rows out. Each row is an independent radio group over `displayCandidates` (keep, then in-range, then minor, then latest, in that display order regardless of `plan`'s emission order), with keep always index 0 and preselected on every row. That preselection is deliberate: the default state of the whole table is a no-op, and there is no select-all or invert, so a major can only be applied by a user deliberately moving the cursor onto its bubble. `↑`/`↓` move the cursor between rows, `←`/`→` move the selection within the row under the cursor, `⏎` applies immediately from wherever the cursor sits and `Esc` cancels writing nothing; both axes clamp at their ends rather than wrapping. The rightmost column shows the peer range that would be written for the row's currently selected bubble (`peerFor`), so the consequence of a selection is visible before committing; for interop rows this is the live group-derived floor from `computeGroupPeers`, and a pick that conflicts with an in-group peer is flagged `⚠`. Long peer-conflict annotations are truncated to the terminal width (`truncateEnd`) so they never wrap and break column alignment. Rows pad to a common cell count and cell width across the whole table so the peer column aligns regardless of how many candidates a given row has.

The radio bubbles are `●`/`○` — the same filled/hollow circles `pnpm up -i` itself uses, replacing an earlier `(•)`/`( )` text rendering. In the table, a selected candidate is colored (green in-range, yellow with the `⚠ major` tag for a major) except a selected **keep**, which is deliberately left uncolored rather than the earlier gray: it is the current value, not a change, and dimming it read as a disabled cell rather than a deliberate no-op. The decision is a pure exported `cellColor(candidate, selected)` in `walk-reducer.ts` rather than inline JSX, because ink-testing-library strips ANSI from its snapshot output — without an exported pure function, a color regression here could not be asserted in a unit test. `summary.ts` picked up the same `●`/`○` glyphs but keeps its own independent style mapping (its `ChangeStyle` vocabulary dims an unselected candidate and a selected keep alike); the static confirmation table and the live one are not required to render pixel-identical, only to agree on what was chosen.

`walk-reducer.ts` is a pure `TableState { cursor, picks, done, cancelled }` machine (`initTable`, `tableStep`, `tableDecisions`); it replaced an earlier `WalkState` that stepped one package at a time. `ui/Walk.ts` is the Ink shell over it. The confirmation summary (`summary.ts`) renders the same grouped, bubble-filled table statically after the walk closes, colored via the existing `ChangeStyle` vocabulary — dim for keep, green for an in-range pick, yellow with a `⚠ major` tag for a major, red for a warning or a rejected edit — so what will be written reads as the same shape the user just chose from.

`edits` (`edits.ts`) turns decisions into `PlannedEdit`s — span replacements tagged with the originating package, whether they rewrite a `range` or a `peer` literal, and the unquoted value. `validate` (`validate.ts`) checks every planned edit's value against the registry before `rewrite` ever sees it: `rangeIsSatisfiable` accepts a range once at least one published version matches it, checked against the ungated version list (see [Command surface](#command-surface)). This is a safety net against a range the pipeline derived wrongly, not a gate on the author's own hand-written ranges, so it fails open (an empty version list or an unparseable range passes) — a derivation the plumbing gets right should never be blocked by a registry hiccup. `validateEdits` rejects ATOMICALLY per package, not per edit: a `strategy` package produces a `range` edit and a `peer` edit as a pair, and if either is unsatisfiable both are dropped together, never just the failing one. Accepting the good half of a pair (say, a satisfiable range bump next to a rejected, stale peer) would leave the file internally inconsistent — the written peer no longer matches what the strategy derives from the newly-written range — and every subsequent run would re-report that drift and re-reject it forever. A package is either fully updated or not touched at all. `rewrite` (`rewrite.ts`) then applies the accepted edits: `applyEdits` sorts descending by start offset so each edit's offsets stay valid as later text shifts, and throws on any overlap. The write is atomic, single-file, formatting-preserving.

## Rationale

### No execution of the config

Discovery and rewrite are 100% static through oxc byte spans. There is exactly one canonical call shape and catalog values are literals, so an "execute for truth" step is unnecessary by construction. This is on-brand for a rolldown-adjacent package and avoids running arbitrary build config to learn its versions.

### Surgical span rewrite, operator preserved

Only the version digits inside a `range` literal change; the leading `^`/`~`/exact operator is reused, so an upgrade never reformats the file or mangles the operator. Complex ranges (`>=5 <6`) and non-literal values are surfaced as skips rather than rewritten. Edits apply right-to-left so spans computed against the original source remain valid.

### Major bumps are interactive-only

`--yes` resolves within range only and never offers or applies the latest-overall/major line. Crossing a major requires running interactively and choosing it; there is deliberately no flag that applies a major automatically.

### Idempotent and re-runnable

The interactive table shows every discovered row, up-to-date rows included as non-selectable context, so a fully up-to-date catalog still renders as the table rather than vanishing; the cursor starts on the first actionable row and the default no-op picks mean re-running and applying writes nothing. Only when discovery finds no catalog packages at all does the command short-circuit with a "nothing to upgrade" message. The earlier `actionableWalkItems` up-to-date filter was removed; `--full` now affects only the non-interactive projection (`projectDecisions`, used by `--preview` and the non-TTY fallback), where up-to-date keeps are omitted unless `--full` is passed.

## Peer handling

`upgrade` is the component that owns peer-range recomputation; the runtime engine never derives (see [architecture.md](architecture.md) and `package/src/catalogs.ts`). `PeerStrategy` has three values: `"lock"`, `"lock-minor"` and `"interop"`. The first two are per-package and offline; `interop` is a per-catalog group reconcile that needs the registry. All three are CLI-only metadata — `normalizeCatalogs` ignores `strategy` and materializes the `peer` into the `<name>Peers` catalog identically regardless.

`derivePeerRange(range, strategy)` (`peer-range.ts`, built on `@effected/semver`) is the single primitive for the offline strategies, operator preserved. It returns `{ range, warning }` (`PeerDerivation`) rather than a bare string, because `lock-minor` can produce a warning alongside its range. `"lock"` pins to the version **as given**, reusing the version text verbatim rather than rebuilding it from parsed `major.minor.patch` — the earlier reconstruction silently dropped prerelease and build identifiers, deriving an unpublished range (`^3.0.0-next.8` → `^3.0.0`) that could never match the very version being catalogued. That bug made `detectPeerDrift` report permanent drift (the derived value could never equal the actual one) and made the keep branch of `buildEdits` rewrite the peer even when the author chose keep — the reported symptom this branch fixes. `"lock-minor"` floors a **stable** version's patch to `.0` (and intentionally drops build metadata in doing so — build metadata identifies a specific build of the un-floored version, and semver ignores it when matching ranges regardless). On a **prerelease** version, flooring is not a meaningful operation (`^3.0.0` excludes `3.0.0-next.8` outright, excluding the very version being catalogued), so `lock-minor` degrades to `lock` behavior and returns a `lock-minor-prerelease` warning rather than silently emitting a range that can never be satisfied. Three behaviors flow from the primitive.

Recompute: on a chosen range bump for an entry with a `strategy`, the candidate's `peerRange` rewrites the existing `peer` literal too, so one upgrade yields two visible edits.

Drift resync (`drift.ts`): if a materialized `peer` no longer matches what `strategy` would produce from the current range (for example a hand-edited range), the walk surfaces a peer-only resync edit even when the range itself is up to date. `detectPeerDrift` explicitly excludes `strategy === "interop"` rather than falling through to the `lock-minor` derivation — an interop peer is computed group-wise from the whole catalog group's cross-peers (see [Interop strategy](#interop-strategy) below), so deriving one per package here would report a resync target that was never correct, surfacing a bogus edit in `--preview`/`--dry-run`.

Materialize (B3): a package with a `lock`/`lock-minor` `strategy` but no `peer` yet gets a new `peer` literal inserted at the end of the range span (`, peer: "<derived>"`), reusing `applyEdits`' zero-width insert support. This happens in both the interactive and `--yes` paths.

### Interop strategy

`interop` exists for a per-catalog group of interrelated packages that declare each other as peers — the motivating case is the `@effect` ecosystem, where members constrain each other through `peerDependencies`. Where `lock`/`lock-minor` derive one package's peer from its own range, `interop` reconciles the whole group against their cross-peers and derives each member's caret-capped peer floor. It generalizes the old `pnpm-plugin-silk` `resolve-effect-versions.ts` helper: group membership (the set of `interop`-marked packages in one catalog) replaces that script's hardcoded `effect`/`@effect/*` filter. The full rationale is in [the interop peer strategy spec](specs/2026-06-27-interop-peer-strategy-design.md).

The group is reconciled by two pure functions in `interop.ts`, fed peerDeps by `runInterop` through an Effectful memoized `FetchPeer` lookup. `resolveGroup` pins each member at its chosen version (the ceiling) and downgrades only dependents to satisfy in-group peers: it never raises a ceiling and never downgrades a peer target, so the dependency-only member (effect core, with no in-group peerDeps) is the de-facto anchor. A member with no satisfiable version at or below its pick becomes an `InteropConflict`, left at the user's pick. `deriveFloors` then sets each member's peer to `^<lowest floor any in-group member declares for it>`, falling back to `^<resolved version>` when nothing peer-depends on it.

The two modes surface conflicts differently. The `--yes`/CI path applies the batch `runInterop` reconciliation once — `resolveGroup` may downgrade dependents below their pick to satisfy in-group peers — and reports any residual conflict. The interactive path no longer downgrades or re-prompts: the live table (`interop-live.ts`, below) recomputes each member's floor and conflict from the user's current picks, and the write path takes those final picks plus the live-derived floors verbatim, reporting whatever conflicts remain. The earlier bounded interactive re-entry loop (`reentryCandidates` re-running `runInterop` per round until the group stabilized) was removed from this path.

The two paths front-load their peer fetches differently. On the `--yes` path `runInterop`'s fetching is lazy: it prefetches only each member's ceiling version (concurrently, `INTEROP_PEER_CONCURRENCY = 8`) to warm the common case where every ceiling is mutually compatible, fetching lower versions on demand inside the downgrade search via the Effectful `FetchPeer`. This replaced an eager prefetch of every candidate version at or below the ceiling, which for a real `@effect` group (dozens of members with hundreds of published versions each) meant thousands of `pnpm view` calls. The interactive path front-loads instead: `buildGroupModel` (`interop-live.ts`) pre-fetches every candidate version's in-group peerDeps up front (concurrently, same `INTEROP_PEER_CONCURRENCY` bound) through a shared memoized `fetchPeer`, then pre-parses every version and range into `SemVer`/`Range` objects so `computeGroupPeers` can recompute floors and conflicts synchronously on each keystroke — parsing is Effectful, comparison and range-testing are not. Both paths share one `(pkg, version)` peerDeps cache, so an immutable lookup is fetched at most once. Because the floor is group-derived, interop always does registry work on a run, even when no range changed — unlike `lock`/`lock-minor`, which detect drift offline.

## Command surface

`rolldown-pnpm-config upgrade [file]` (`commands/upgrade.ts`). The file arg is optional: when omitted, `select-file.ts` scans the cwd for a `.ts` file containing a usable `PnpmConfigPlugin(...)` call and errors on zero or multiple candidates. Flags: `--yes`/`-y` (non-interactive latest-in-range), `--dry-run` (run the same flow as an unflagged run and skip only the final write), `--catalog <name>` (restrict to one catalog), `--preview` (non-interactive projection: discover → resolve → plan → take default picks → print colored summary → exit; no walk, no write) and `--full` (include up-to-date packages in the non-interactive projection printed by `--preview` and the non-TTY fallback). The default path is the interactive walk, which now shows every discovered row — up-to-date rows as non-selectable context — with the cursor starting on the first actionable row; the earlier `actionableWalkItems` filter was removed and `--full` no longer affects the interactive table. A confirmation diff (`summary.ts`, now returning `StyledLine[]` and rendered via the shared `toAnsi`) is printed before any write; it reports any remaining `InteropConflict`s alongside the range/peer edits. Output is colorized when the terminal supports it (detected via `ui/env.ts`).

Non-TTY/CI fallback: when the terminal is not interactive (`hasTTY && !isCI && !isAgent` is false), the command automatically skips the walk and prints the non-interactive projection instead of entering raw-mode and hanging. This is gated on `capabilities.interactive` from `cli/ui/env.ts`.

`--preview` vs `--dry-run`: `--preview` never prompts (runs the full pipeline, takes the default picks, then exits). `--dry-run` is not a separate code path — it runs the identical interactive flow (table → picks → interop reconcile → validate → summary) and skips only the final `writeFileSync`. It used to short-circuit before the table, printing a summary of the auto-picked defaults the author never chose and skipping the interop reconcile entirely, so the "preview" did not match what an apply would actually do; running the real flow makes the two modes' output structurally identical except for the write. The table's header carries a `DRY RUN — nothing will be written to the config` banner in that mode, and the run ends with a `Dry run — no changes written. N change(s) would be applied.` line instead of `Applied N change(s).`, so the mode is legible in both the interactive UI and the final output.

The `minimumReleaseAge` gate applies on every run regardless of flag — both `--yes` and the interactive walk resolve only against versions old enough for the author's own pnpm install to accept (see [Pipeline](#pipeline)). `resolveGatedVersions` returns both the gated list (the only candidate source) and the raw, ungated list; validation always checks against the raw list, since a candidate the gate is hiding for freshness is still perfectly satisfiable.

Every planned edit is validated against the registry (`validate.ts`) before it reaches `rewrite`, regardless of mode. The two modes diverge on what a rejection means. Interactively, a rejection is dropped and reported in the confirmation summary at the granularity of a whole package (both its `range` and `peer` edits, per the atomicity described above), not a single edit; the rest of the run proceeds, so one bad package cannot block an otherwise-good upgrade. `--yes` is strict instead: it fails hard (non-zero exit, nothing written) on **any** warning, whether a `lock-minor`-on-a-prerelease incompatibility from `derivePeerRange` or an edit `validateEdits` found unsatisfiable. The asymmetry is deliberate — an interactive user sees a warning and can go fix the config; a `--yes` run is unattended (CI), where a warning scrolling past unread is how a broken range reaches a published artifact. `--yes` continues to never apply a major regardless of warnings, unchanged from before this branch.

Note: the brainstorm spec mentions an `up` alias and a `-i/--interactive` flag; the as-built command exposes neither (interactive is the unflagged default). Treat this doc as authoritative on the shipped surface.

## Load-bearing types

`CatalogEntry`, `Candidate` and `Edit` in `package/src/cli/types.ts` are the values that cross the discover → plan → rewrite boundaries (`Candidate.kind` is now `"keep" | "in-range" | "minor" | "latest"` — the `minor` tier is the same-major intermediate `plan` adds); `WalkItem` and `Decision` in `package/src/cli/walk-types.ts` are the interactive-layer contract between `walk-plan`, the reducer and `edits`/`summary`. `PeerStrategy` (now `"lock" | "lock-minor" | "interop"`) and `CatalogPackageSpec` are shared with the authoring surface in `package/src/catalogs.ts`.

`PeerDerivation` (`peer-range.ts`) is `derivePeerRange`'s return shape — `{ range, warning }` — and `PeerWarning` (`{ kind: "lock-minor-prerelease", message }`) is the non-fatal incompatibility it can carry; both are load-bearing because the warning has to propagate all the way to `WalkItem.peerWarning`, the table row, the summary and `--yes`'s strictness check. `PlannedEdit` (`types.ts`) extends `Edit` with `pkg`, `kind: "range" | "peer"` and the unquoted `value` — provenance `buildEdits` attaches so `validateEdits` (`validate.ts`) can check each edit's value against the registry before `applyEdits` (which still takes the plain `Edit` shape) ever sees it; `RejectedEdit` (`validate.ts`) is what a failed check produces, threaded into the summary and (under `--yes`) the fatal error message. `TableState` (`walk-reducer.ts`) — `{ cursor, picks, done, cancelled }` — is the interactive table's state, one radio-group `pick` per row; it replaced the older per-package `WalkState`.

The interop pass adds its own load-bearing values in `package/src/cli/interop.ts`: `GroupMember` (a member's pkg, ceiling and candidate versions) is the input to `resolveGroup`; `InteropConflict` (an unsatisfiable member with its blocking constraints) and `InteropResult` (the resolved versions, derived peers, conflicts and the fetch-cache lookup) are what the reconcile produces and what the command and `summary` consume. `FetchPeer` (`(pkg, version) => Effect<Record<string, string>, never>`) is the Effectful memoized peer-lookup contract `resolveGroup`/`violations`/`deriveFloors` take for the internal lazy resolve path, backed by the shared cache; the synchronous `PeerDepsOf` is now only the `InteropResult.peerDepsOf` ceiling reader. `ReleaseAgeGate` and `PartialReleaseAgeGate` are imported from `@effected/npm` (the kit owns the gate model, not this repo): `ReleaseAgeGate` is the combined gate every candidate list is filtered against, built by `computeGate` (`commands/upgrade.ts`) from the two local `PartialReleaseAgeGate` contributions that `readConfigReleaseAge`/`parsePnpmGate` (`package/src/cli/release-age.ts`, the only release-age logic still local) derive.

The live interactive interop path adds its own load-bearing values in `package/src/cli/interop-live.ts`: `GroupModel` (the pre-fetched, pre-parsed model of one catalog group — in-group `peerReqs` per `pkg@version`, plus parsed `SemVer`/`Range` maps) is what `buildGroupModel` produces and `computeGroupPeers` consumes, and `GroupPeers` (`{ peer, conflict }` maps keyed by member) is the synchronous per-pick recompute the table renders. `buildGroupModel` returns `Effect<GroupModel, never>`; `computeGroupPeers(model, selected)` is pure. `truncateEnd` in `walk-reducer.ts` clips a long peer-conflict annotation to the terminal width.

Two concurrency bounds are load-bearing tuning constants: `RESOLVE_CONCURRENCY` (12) in `commands/upgrade.ts` caps the per-package version+times fan-out, and `INTEROP_PEER_CONCURRENCY` (8) in `interop.ts` caps the ceiling-prefetch fan-out.
